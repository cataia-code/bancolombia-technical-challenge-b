Prueba técnica · Senior Software Engineer · Parte B

Agente de inteligencia para consolidación y priorización de migración de taskbots

Una solución local y reproducible que recibe un inventario fragmentado de taskbots (RPA) y produce decisiones accionables: qué consolidar, a qué destino migrar (n8n / microservicio / Python-Java / RPA selectivo), qué requiere gate de gobierno/IA y qué queda en revisión manual profunda — con reglas explicables, trazabilidad y sin depender de ambientes del cliente.

Resumen ejecutivo

El reto no es "hacer un bot": es demostrar criterio de diseño para convertir una masa de automatizaciones fragmentadas en decisiones defendibles. La solución detecta duplicidad, clasifica el destino tecnológico por reglas explicables y prioriza por valor vs. complejidad — todo ejecutable en el entorno local y orquestable con n8n.

Se apoya en tres decisiones nucleares, alineadas con la propuesta de la Parte A:

1 · Racionalizar antes de migrar

De un inventario plano a un catálogo de utilidades: las variantes se consolidan; no se migra una por una.

2 · Orquestar, no bot-izar

n8n coordina el análisis vía un servicio con API; la lógica de negocio vive en el servicio, no en el orquestador.

3 · Criterio explicable

Reglas deterministas deciden y justifican; un agente LLM opcional enriquece la redacción sin cambiar la decisión.

Qué se considera una solución correcta

Una solución que (a) corre localmente y es reproducible (mismas entradas → mismas decisiones), (b) demuestra reducción de fragmentación mediante consolidación, (c) es trazable y auditable, y (d) explica por qué recomienda cada destino y cada ola — sin sobre-ingeniería.

Parte B · Entendimiento del reto

El problema y sus requisitos

Migrar desde una plataforma de RPAs hacia una arquitectura evolutiva exige decidir qué se migra, qué se consolida y qué se reemplaza — y demostrar valor sin depender de ambientes del cliente.

Problema

Un portafolio de RPA crece de forma fragmentada: decenas de taskbots que se duplican, dependen de UI legacy y mezclan integraciones. Decidir manualmente qué hacer con cada uno es lento y subjetivo. Esta solución automatiza el criterio.

Entrada

Inventario con nombre, propósito, aplicaciones, tipo(s) de interacción, frecuencia, riesgo, dependencias y evidencia de duplicidad. Formatos: .txt, .csv, .json, .sqlite.

Salida

Por cada taskbot: destino sugerido, ola, valor/complejidad y justificación; más grupos consolidables y un reporte JSON + HTML.

Requisitos funcionales

IDRequisitoCriterio de aceptación

Requisitos no funcionales

Reproducibilidad

Mismas entradas → mismas decisiones. El LLM solo afecta la redacción.

Testabilidad

Dominio puro sin I/O; pruebas unitarias e integración en CI.

Bajo acoplamiento

Puertos y adapters: fuente, salida, LLM y orquestador intercambiables.

Extensibilidad

Nuevos destinos/reglas/fuentes sin tocar el núcleo.

Observabilidad

Logs JSON con correlation id por corrida.

Portabilidad

Local y Docker; sin credenciales del cliente.

Supuestos y preguntas abiertas

Supuestos

La duplicidad suele estar declarada en el inventario; el tipo de interacción refleja la vía de integración real; la UI legacy es el factor bloqueante de migración (criterio conservador → RPA selectivo).

Preguntas al evaluador

¿Existe volumen/criticidad por bot para afinar el valor? ¿Hay un catálogo de APIs existentes que permita reclasificar legacy con API disponible como candidato a n8n?

Diseño · Bloque A

Arquitectura hexagonal ligera

El criterio de negocio (reglas, scoring, clustering) es puro y estable; las orillas (fuente, salida, LLM, HTTP) son intercambiables. Un solo despliegue, sin microservicios injustificados.

Alineación con la Parte A

La Parte A propuso racionalizar antes de migrar, orquestar en lugar de bot-izar y gobierno desde el diseño. Esta Parte B es la implementación local y demostrable de ese criterio: el discovery de la Parte A (Jaccard + union-find + scoring) evoluciona aquí a un servicio hexagonal con destinos, olas, orquestación n8n y agente.

Trade-offs aceptados

Puerto entrenable explícito

La calibración de "apps hub" pasa por TrainableSimilarityScorer; los scorers simples solo implementan SimilarityScorer.

rapidfuzz como dependencia

Mejor similitud en nombres en español a cambio de una dependencia ligera; la idea de clustering es la misma que la Parte A (stdlib).

Alternativas descartadas

Microservicios

No aplica: un solo dominio, un solo despliegue. Fragmentaría sin beneficio.

Capas clásicas

Acoplan dominio a persistencia; hexagonal invierte esa dependencia.

LLM como núcleo

Rompe reproducibilidad y la restricción de correr sin credenciales.

Diseño · Bloque B

Modelo de dominio y reglas de negocio

Reglas deterministas y explicables: la primera que aplica gana, y cada una emite su razón. El eslabón frágil (UI legacy) domina el destino.

Reglas de clasificación de destino

#CondiciónDestinoPor qué
1Sin interacción reconocidaRevisión manualNo se puede clasificar destino
2Presencia de UI legacyRPA selectivoSin integración limpia; el eslabón frágil manda
3Variante en cluster grande (≥3, sin legacy)MicroservicioUtilidad reutilizable compartida
4Presencia de base de datosPython/JavaLógica de datos a la medida

Gate de gobierno, no cola manual masiva

ReviewPlan es independiente del destino: un taskbot puede ir a n8n, microservicio o RPA selectivo y aun asi requerir prechequeo IA, aprobacion dirigida o evaluacion manual profunda. La IA no cambia la decision; prepara evidencia y checklist para reducir trabajo humano.

Scoring y ola

Valor

Frecuencia (0–40) + riesgo mitigado (0–30) + bono de consolidación (0–30). Mayor frecuencia y duplicidad ⇒ mayor valor.

Complejidad

Máx. complejidad de interacción (UI legacy domina) + canales extra + riesgo + dependencias (tope 20).

OlaCondiciónInterpretación
Ola 1valor ≥ 70, complejidad ≤ 65 y sin gate de gobiernoQuick wins seguros
Ola 2valor ≥ 50 y complejidad ≤ 85Consolidación / habilitación controlada
Ola 3restoBajo valor relativo o complejidad extrema

Detección de duplicidad multi-señal

El clustering (union-find) combina tres señales, evitando el encadenamiento del single-linkage al excluir apps "hub" (presentes en ≥25% del portafolio, p.ej. SAP ECC):

1 · Evidencia declarada

"Similar a TB_X" en el inventario: la señal más fuerte y confiable.

2 · Solapamiento de apps

Jaccard sobre aplicaciones no-hub: captura duplicidad no declarada.

3 · Similitud textual

rapidfuzz sobre nombre + propósito.

Hallazgo de negocio

El 72% del portafolio depende de UI legacy. La estrategia realista no es forzar n8n, sino: (1) consolidar variantes, (2) exponer APIs donde hoy se navega UI para habilitar migración futura, y (3) reservar RPA selectivo para lo que aún no tiene integración.

Por qué Ola 3 queda acotada

Los 27 gates de gobierno no son 27 evaluaciones manuales completas: 19 se resuelven con prechequeo IA o aprobación dirigida y solo 8 quedan como evaluación manual profunda. Por eso los legacy de alto valor pasan a Ola 2 como habilitación/consolidación controlada y Ola 3 queda para rediseño o baja prioridad relativa.

De recomendaciones a plan de racionalización

La salida no es una lista de recomendaciones sueltas: es un plan accionable con piezas que el reporte genera automáticamente.

Catálogo de componentes

15 componentes reutilizables extraíbles: nombre sugerido, miembros, propósito común, patrón destino, apps dominantes, blocker legacy y acción recomendada. Conecta con "qué componentes reutilizables extraer".

Matriz API / no-API

Por sistema y por operación: api_disponible, api_requerida, bloqueador, acción habilitadora y destino_objetivo_post_habilitacion. Hace defendible el 72% legacy: no es resignación a RPA, es un plan de API enablement.

Política de revisión

El reporte separa gates de gobierno, prechequeo IA, aprobación dirigida y evaluación manual profunda para reducir trabajo humano sin perder control.

Evidence pack

Cada gate trae dependencias, controles, checklist, owner sugerido, bloqueadores y siguiente acción. Esto vuelve auditable la intervención de IA antes de la aprobación.

Sensibilidad de umbrales

El reporte recalcula gates, manual profunda y olas con complejidad 80/85/90 y dependencias 3/4 para mostrar si la clasificación es robusta.

Scoring explicado

Cada recomendación trae su score_breakdown: frecuencia, riesgo, duplicidad, complejidad por interacción y dependencias. Permite defender por qué algo cayó en Ola 1/2/3.

Guardrails de seguridad

El endpoint /analyze valida run_id (regex), contiene el inventario dentro de TASKBOT_INVENTORY_ROOT y recomienda /analyze/inline para exposición externa. Ver arquitectura y recuperacion.

Operación · Bloque D

Ejecución local y orquestación con n8n

Se demuestra por CLI (sin n8n) o por n8n (orquestada). El servicio expone un contrato HTTP simple; el orquestador es reemplazable por Appian/Power Platform/BPM.

Demo en un comando

# desde la raíz del repo
bash scripts/demo.sh          # Linux/macOS/Git-Bash
.\scripts\demo.ps1            # Windows PowerShell

CLI

# instalar una vez (desde la raíz)
python -m venv .venv && . .venv/Scripts/activate   # o source .venv/bin/activate
pip install -e "./poc[dev]"

# analizar el inventario real
python -m taskbot_advisor analyze poc/data/ejemplo_50_taskbots_prueba.txt
# → reports/<run_id>/reporte.json  +  reporte.html

API + n8n (local)

cd poc && docker compose up --build
# API   → http://localhost:8000  (/health, /analyze, /analyze/inline)
# n8n   → http://localhost:5678  → Import from File → n8n/workflow.json

Al ejecutar Test workflow, el nodo final debe devolver run_id, headlines, resumen y ola_3 con el desglose de causas.

Rollback / recuperación

El servicio es stateless y de solo lectura sobre el inventario. Cada corrida es idempotente y versionada por run_id en reports/<run_id>/ sin sobrescribir. Fail-soft por ítem; el nodo HTTP de n8n reintenta 3×. Recuperarse = volver a ejecutar.

Decisiones

Architecture Decision Records

Cinco decisiones resumidas en un unico documento tecnico.

ADR-001 · Arquitectura hexagonal

Núcleo puro y testeable; orillas intercambiables. Sin microservicios.

ADR-002 · Reglas deterministas

Clasificación explicable y reproducible; una razón por regla.

ADR-003 · Similitud multi-señal

Declarada + apps no-hub + texto; evita encadenamiento single-linkage.

ADR-004 · Agente híbrido con fallback

Reglas deciden; LLM opcional solo redacta y degrada limpio.

ADR-005 · n8n orquestador desacoplado

Lógica en el servicio HTTP; n8n orquesta. Mapeable a Appian/Power Platform.
